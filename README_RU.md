# VESC Express

Кодовая база для VESC Express — логгера и IO-платы с WiFi и Bluetooth. На данный момент протестировано и работает на ESP32C3 и ESP32S3, но можно добавить и другие ESP32.

## Тулчейн

Инструкция по настройке тулчейна:
[https://docs.espressif.com/projects/esp-idf/en/latest/esp32c3/get-started/linux-macos-setup.html](https://docs.espressif.com/projects/esp-idf/en/latest/esp32c3/get-started/linux-macos-setup.html)

**Примечание**
Для сборки проекта требуется ESP-IDF версии 5.2 или выше.

### Установка релиза 5.2.2

Инструкция по ссылке выше установит master-ветку ESP-IDF. Для установки стабильного релиза перейдите в директорию установки и выполните:

```bash
git clone -b v5.2.2 --recursive https://github.com/espressif/esp-idf.git esp-idf-v5.2.2
cd esp-idf-v5.2.2/
./install.sh esp32c3 esp32s3
```

На данный момент разработка ведётся на стабильном релизе 5.2.2.

## Сборка

Задайте целевой чип/архитектуру:
```bash
idf.py set-target <target>
```

где target — esp32c3 или esp32s3. При смене таргета необходимо выполнить fullclean или удалить директорию build.

После настройки тулчейна проект собирается командой:

```bash
idf.py build
```

Это создаст vesc_express.bin в директории build, который можно использовать с бутлоадером в VESC Tool. Если на ESP32c3 нет предустановленной прошивки, можно прошить через USB-порт используя встроенный бутлоадер. Для этого также потребуются bootloader.bin и partition-table.bin из директории build. Прошивка возможна из VESC Tool или через idf.py.

Сборка всех таргетов:

```bash
python build_all.py
```

Это создаст все необходимые файлы прошивки в директории build_output с поддиректориями по именам железа. Переключение таргетов выполняется автоматически.

### Кастомные конфигурации железа

Для сборки с кастомными конфигурациями положите файлы конфигурации в директорию "**main/hwconf**" и используйте флаг HW_NAME:
```bash
idf.py build -DHW_NAME="VESC Express T"
```

**Примечание:** При изменении переменных окружения (или при первом использовании) необходимо сначала выполнить `idf.py reconfigure` перед сборкой, так как система сборки не может автоматически обнаружить это изменение. `idf.py fullclean` даёт тот же эффект.

### Скрипт сборки для Windows

Скрипт `build_idf.ps1` для Windows автоматически настраивает переменную окружения MSYSTEM и PATH для Python.

```powershell
.\build_idf.ps1 build                              # сборка с HW по умолчанию
.\build_idf.ps1 -DHW_NAME="sodovaya express" build  # сборка с кастомным HW
.\build_idf.ps1 fullclean                            # очистка сборки
.\build_idf.ps1 flash                                # прошивка через USB
```

Переменные окружения (опционально):
- `IDF_PATH` — путь к установке ESP-IDF (по умолчанию: `C:\esp\esp-idf-v5.2.2`)
- `PYTHON3_DIR` — путь к директории Python 3.10+ (по умолчанию: `%LOCALAPPDATA%\Programs\Python\Python310`)

## Мост ANT BMS

Этот форк добавляет BLE GATT Client мост, который подключается к ANT BMS, читает данные батареи и пересылает их в экосистему VESC в виде стандартных BMS CAN-сообщений. VESC ESC видит его как нативную BMS — напряжения ячеек, температуры, ток, SOC — всё отображается на странице BMS в VESC Tool.

Идея этого моста вдохновлена [kamiyamaQ](https://www.youtube.com/@kamiyamaQ).

**ВНИМАНИЕ:** Это экспериментальная кастомная прошивка. Она может быть нестабильной, вызывать непредвиденное поведение или окирпичить устройство. Прошивайте на свой страх и риск. Авторы не несут ответственности за любой ущерб вашему оборудованию.

**Протестировано только с [Flipsky FSESC Express](https://flipsky.net/collections/fsesc-express/products/flipsky-fsesc-express).** Другое железо не тестировалось.

### Как это работает

ESP32-C3 одновременно работает с тремя радиоинтерфейсами:
- WiFi — подключение VESC Tool (TCP)
- BLE Peripheral (GATTS) — подключение мобильного VESC Tool
- BLE Central (GATTC) — подключение к ANT BMS

При загрузке (если включён) Express сканирует настроенный MAC-адрес ANT BMS, подключается по BLE, подписывается на уведомления характеристики 0xFFE1 и опрашивает статус каждые 2 секунды. Полученные данные проверяются, записываются в структуру VESC `bms_values` и отправляются по CAN с адаптивным ограничением скорости, уступая трафику пересылки конфигурации.

### Пересылаемые данные

```
Напряжения ячеек (до 32)   u16LE милливольты @ смещение 34
Общее напряжение            u16LE * 0.01V
Ток                         i16LE * 0.1A
SOC                         u16LE (%)
SOH                         u16LE (%)
Температуры ячеек           u16LE прямые °C (65496 = нет датчика)
Температура MOS             u16LE прямые °C  ->  Temp PCB в VESC Tool (макс. из MOS и балансира)
Температура балансира       u16LE прямые °C  ->  Temp Hum в VESC Tool (MOS темп. отдельно)
Ключи заряда/разряда        u8 флаги
Состояние балансировки      u16LE битовая маска (1 бит на ячейку)
Остаточная ёмкость          u32LE * 0.000001 Ah  ->  Ah counter + Wh counter в VESC Tool
```

### Команды терминала

Подключитесь к Express через VESC Tool > VESC Dev Tools > Terminal:

```
ant_bms                            показать статус: включён, MAC, состояние, напряжение, ток,
                                     SOC, SOH, ячейки, температуры, Temp PCB (MOS/BAL разбивка),
                                     остаточные Ah/Wh, состояние балансировки, разрешение заряда
ant_bms enable                     включить мост (запускает BLE сканирование, сохраняется в NVS)
ant_bms disable                    выключить мост (отключает BLE, очищает данные BMS, сохраняется в NVS)
ant_bms mac XX:XX:XX:XX:XX:XX     сменить MAC целевой BMS (сохраняется в NVS, применяется после перезагрузки)
ant_bms poll 500                   сменить интервал опроса в мс (500-10000, применяется сразу, сохраняется в NVS)
```

### Конфигурация

Все настройки хранятся в NVS flash и сохраняются при перезагрузках и OTA-обновлениях.

Мост включён по умолчанию. MAC по умолчанию — `C3:5E:E2:61:94:AE` (захардкожен в `ant_bms.c`). Интервал опроса по умолчанию — 2000мс. С одновременно работающими WiFi + BLE Peripheral + BLE Central опрос 500мс протестирован стабильно на ESP32-C3, но 2000мс — безопасное значение по умолчанию.

**ВНИМАНИЕ: Запись конфигурации ESC при активном логгере и BMS мосте может вызывать ошибки "Parameters truncated". Частично записанная конфигурация может оставить ESC в нерабочем состоянии (мотор не крутится) — восстанавливается перезагрузкой, или потенциально необратимо, если повреждённые значения затронут критичные настройки.** BMS мост автоматически уступает CAN шину при пересылке конфигурации, но логгер генерирует CAN трафик со стороны ESC, который Express не может приостановить. Для надёжной записи конфигурации: остановите логгер и выполните `ant_bms disable`. Обычная езда (без записи конфигурации) должна быть безопасна при всех активных функциях.

### Конфигурация железа

Конфигурация `hw_sodovaya` основана на `hw_xp_t` (VESC Express T) с тем же пинаутом. Для сборки:

```bash
idf.py build -DHW_NAME="sodovaya express"
```

### Механизмы безопасности

- **Валидация данных**: все полученные значения проверяются на допустимый диапазон перед записью в `bms_values` (напряжение vs количество ячеек, ток, температуры, SOC/SOH). Невалидные фреймы отбрасываются целиком.
- **Уступание CAN при пересылке конфига**: BMS фреймы подавляются пока `comm_can_send_buffer()` отправляет многокадровый конфигурационный пакет (через флаг `comm_can_is_forwarding()`). Дополнительно каждый BMS CAN фрейм проверяет свободность шины перед отправкой и прерывает серию при обнаружении трафика. Нет таймаута принудительной отправки — ESC обрабатывает устаревание данных через собственный 2-секундный `MAX_CAN_AGE_SEC` (fail-open).
- **Watchdog устаревших данных**: если подключён, но нет валидного парсинга 30 секунд (BMS перестала отвечать), принудительно отключает BLE и переподключается.
- **Очистка при отключении**: при разрыве BLE соединения `bms_values.update_time` обнуляется, ESC по таймауту снимает ограничения тока по BMS (fail-open дизайн).

### Заметки по протоколу ANT BMS

- BLE сервис 0xFFE0, характеристика 0xFFE1 (notify + write-no-response)
- Фрейм команды: `7E [payload] [CRC16-MODBUS LE] AA 55` — CRC считается только по payload (без заголовка 0x7E)
- Фрейм ответа: `7E A1 [code] XX XX [data_len] [data...] [CRC16 LE] AA 55`
- Все многобайтовые поля — little-endian
- Код ответа статуса: 0x11
